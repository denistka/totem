# Правила использования codemap для агента

Сгенерированные JSON в этой папке — **многоуровневая карта** клиентского пакета (`src-client`). Цель: получить структуру и связи **без чтения исходников**, экономя контекст; открывать код только там, где карты недостаточно.

Обновление артефактов: из корня `src-client` выполнить `pnpm codemap` (также запускаются `jscpd` и проверка циклов `madge`).

---

## Порядок «масштаба» (сначала грубо, потом детальнее)

1. **`codemap-src-paths.json`** — только отсортированные пути под `src/`. Нужен для: оценки раскладки по папкам, поиска аномалий в именовании/вложенности, быстрого списка файлов.

2. **`codemap-surface.json`** — для каждого `src/**/*.ts(x)`:
   - экспортируемые имена и их **вид** (`function`, `class`, `const`, `re-export`, `default`, …);
   - **`referencesFromOtherFiles`** — сколько раз на модуль ссылаются (все учтённые виды импорта);
   - **`importsFromProjectTs` / `importsFromProjectTsCount`** — **уникальные** зависимости от других TS/TSX файлов в `src` (без CSS/JSON, чтобы не зашумлять граф кода).

   В начале файла есть **`mapGuide`** с кратким описанием уровней.

3. **`codemap-import-refs.json`** — каждое подключение: из какого файла, **номер строки**, вид (`import`, `export-from`, `dynamic-import`, `require`), текст specifier. Нужен, когда важно **где именно** тянут модуль или для точечного рефакторинга импортов.

4. **`codemap.json`** — дерево репозитория клиента с метриками (строки, байты, кодпоинты) по файлам. Папка `codemap/` из обхода **исключена**. Нужен для оценки размера файлов и полной структуры вне только `src/`.

---

## Правила работы агента

- **Сначала** определить задачу на карте: пути → при необходимости surface → при необходимости import-refs → при необходимости полное дерево `codemap.json`.
- **Не читать исходники пакетно** для ответов вида «где лежит X», «кто импортирует Y», «сколько входящих связей у модуля» — сначала свериться с JSON.
- **Высокая `referencesFromOtherFiles`** у маленького по смыслу модуля — кандидат на «узел» при рефакторинге: менять контракт дороже.
- **Большой `importsFromProjectTsCount`** — сильная связность с остальным `src`; при переносе/разбиении модуля планировать каскад правок.
- **Barrel-файлы** часто дают много **`re-export` / `re-export-type`** в `exports` — это нормально; смотреть сочетание с входящими/исходящими рёбрами.
- **`unresolved`** в `codemap-import-refs.json` (если есть) — локальные пути, которые анализатор не смог сопоставить с файлом; для точных правок может понадобиться открыть файл.
- Карта **статическая**: нет рантайма, условных `import()` с нелитеральным путём, алиасов вне `tsconfig.app.json` / Vite. Внешние пакеты (`node_modules`) в граф **не входят** как цели.

---

## Ограничения

- Экспорты собираются эвристикой по AST; редкие конструкции могут отличаться от того, как бандлер собирает entry.
- Исходящие рёбра в surface считаются только до **TS/TSX** в `src`; импорты стилей и JSON учитываются в import-refs при разрешении, но не попадают в `importsFromProjectTs`.

## Файлы в этой папке

| Файл                       | Роль                                               |
| -------------------------- | -------------------------------------------------- |
| `codemap-src-paths.json`   | Уровень 1: пути                                    |
| `codemap-surface.json`     | Уровень 2: экспорты + счётчики и локальный TS-граф |
| `codemap-import-refs.json` | Уровень 3: все места импорта с строками            |
| `codemap.json`             | Уровень 4: дерево + размеры                        |
| `AGENT-GUIDE.md`           | Этот документ (не генерируется скриптом)           |
